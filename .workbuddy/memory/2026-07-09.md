## WDM 驱动死机分析 (go2cbt.sys)

**任务**: 分析 Windows WDM 框架磁盘写钩子驱动服务器死机原因

**发现的问题**:
- **P0 致命**: 缺少 IoCreateDevice 调用，导致 NULL 指针解引用 (go2cbt.c:21,48-49)
- **P1 严重**: 设备创建顺序错误，IoCreateSymbolicLink 在设备创建前调用
- **P2 高危**: HwReadWrite 函数存在无限递归风险 (write.c:16,56)
- **P3 中等**: 全局数组无锁保护，KdPrint 在高 IRQL 不安全

**结论**: 驱动加载时即触发 BSOD (0x00000050)，根因是 DriverEntry 未创建控制设备对象

必须同时修复 hook.c (取消注释) 和 write.c (修复所有递归漏洞) 才能稳定运行

## 第四次检查 (16:48) - 用户质疑后的重新深度分析

### 用户正确指出的逻辑问题
- **Bug #1 (write.c:16) 无效**: 只Hook了WRITE，非WRITE请求不应进入HwReadWrite
- **Bug #2 (write.c:56) 无效**: WRITE请求走到这里，设备肯定已被Hook且在g_HookList中

### 🔴 重新定位的真正死机根因（按概率排序）

**🥇 #1 KdPrint 性能灾难（最可能！）**
- 服务器磁盘 I/O = 数百~数千 IOPS
- 每次写操作都调用KdPrint（前100次）
- KdPrint 需要获取内核调试缓冲区自旋锁
- 多核CPU并发争用此锁 → 所有核心阻塞 → 系统冻结

**🥈 #2 IRP Pending 处理缺失**
- write.c:93 直接 return originalWrite(DeviceObject, Irp)
- 未检查 STATUS_PENDING 状态
- 异步完成的IRP可能导致双重完成或内存损坏

**🥉 #3 IRQL/线程安全问题**
- KdPrint 在 DISPATCH_LEVEL 以上不安全
- g_DiskMap/g_HookList 全局数组无锁保护
- 多核数据竞争 → ABA问题或崩溃

## 第六次进展 (18:00) - TEST2-3.log 日志深度分析

### 日志统计
- **总行数**: 7,143 行 (1.1MB)
- **运行时间**: 927.3 秒 (~15.5 分钟)
- **Unknown Device 写入**: 6,966 条 (**97.7%**) ← 核心问题！
- **已追踪写入**: Part0=50条, Part1-5=50条 (共 100 条 = 1.4%)
- **系统日志**: 14 行 (初始化 + 卸载)

### 🎯 根因定位：Unknown Device = VolMgr 卷设备

**关键证据**:
```
Unknown: DevObj=0xFFFFA3835419F9E0   DrvObj=0xFFFFA38352F770A0
Part0:    DevObj=0xFFFFA38354186040   DrvObj=0xFFFFA38352F770A0  ← 同一个 DrvObj!
```

**结论**: `0xFFFFA3835419F9E0` 是 Volume Manager 为 Disk0 创建的**逻辑卷设备对象**（如 `\Device\HarddiskVolumeX`），不是物理分区。`BuildDiskAndHookTables()` 只枚举 `\Device\HarddiskX\PartitionY`，所以永远不会发现它。

### 设备栈层次
```
文件系统驱动 (NTFS/FAT)
  └─ VolMgr 卷设备 Dev=FFFFA3835419F9E0 ← Unknown! (97.7%写操作走这里)
      └─ PartMgr 分区设备 Dev=FFFFA383541B2040 (Part3)
          └─ Disk.sys 物理磁盘 Dev=FFFFA38354186040 (Part0/MBR)
```

### 写入模式（关键铁证！）
- **每个 Unknown 写入都与 Part3 写入逐字节一致**（相同 offset + length）
- 严格的时间顺序：Part3 → Unknown(+10~72μs) → Part0
- Part0 offset = Part3 offset + 0xD90000 (固定分区起始偏移)
- I/O 转发链: NTFS → VolMgr(Unknown) → PartMgr(Part3) → Disk.sys(Part0)

## 第七次进展 (18:55) - 最终结论：不需要劫持！

### 核心判断依据（来自 TEST2-3.log 数据）

| 时间(秒) | 设备 | Offset | Length | 说明 |
|----------|------|--------|--------|------|
| 0.93489021 | **Part3** ✓ | `0x17f3a000` | `0x2000` | CBT 已追踪 |
| +72μs | **Unknown** ✗ | `0x17f3a000` | `0x2000` | 与 Part3 完全重复! |
| +21μs | **Part0** ✓ | `0x2583a000` | `0x2000` | CBT 已追踪 (+偏移) |

**Unknown 设备的每一笔写入都被底层 Part3/Part0 完整捕获，不存在 CBT 遗漏！**

### 最终结论（经 WinDbg 验证 19:34）
- **Unknown 设备 = partmgr.sys 创建的内部卷级设备对象** (Dev=5419F9E0, Drv=\Driver\partmgr)
- **关键特征**: `!devstack` 只有自己一行，**无下层设备栈** (无 AttachedTo)
- 不是物理 I/O 目标，是 partmgr 的 IRP 转发代理
- partmgr 共 7 个 DevObj，其中 5419F9E0 是唯一无下层栈的幽灵设备
- **不需要劫持**——每笔写入都 100% 镜像到已追踪的 Partition
- **只需删除 write.c:47 的 KdPrint 日志即可**（占 97.7% 日志量）

## 第八次进展 (21:43) - 关键架构简化：只监控 Part0

### 新日志数据 (TEST2-3.log 最新版, 143行)

| 指标 | 值 |
|------|-----|
| **Part1-5 写入** | 50 条 |
| **Part0 写入** | **50 条** (完美 1:1!) |
| **Unknown 设备** | **0 条** (彻底消失) |
| **DiskOff 一致性** | **100% 逐字节相同** |

### 核心发现：PartN 的每笔写入都在 Part0 有完全相同的 DiskOff
- 不需要 g_DiskMap[] 分区表，不需要 PartitionStartingOffset 偏移转换
- 只 Hook Part0 的 disk.sys DriverObject 即可覆盖所有物理块变更
- RTL_BITMAP 直接用 DiskOff / CBT_BLOCK_SIZE 作为 bit 索引

### 技术讨论
- **CBT 数据结构选型**: 推荐 RTL_BITMAP（固定内存、去重、行业标准），不推荐 SLIST（内存无界增长）
- **分区信息获取**: 编写了 QueryPartitionInfoEx() 函数，通过 IOCTL_DISK_GET_PARTITION_INFO_EX 获取 StartingOffset + PartitionLength
- **卸载问题**: 发现 CbtUnload 缺少 IoDeleteSymbolicLink 和 IoDeleteDevice 调用，导致 sc stop 后无法删除驱动文件
